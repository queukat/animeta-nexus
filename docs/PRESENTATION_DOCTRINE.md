# Presentation Doctrine

AniMeta Nexus is presented as a high-rank metadata control system.

The rule is simple:

> Do not describe what the code mechanically does. Describe the domain of chaos
> each capability brings under control.

The implementation can be practical, compact, and honest. The presentation must
make every capability feel named, ranked, and operationally significant.

## Capability Naming Rule

Do not say:

- script
- helper
- simple translator
- API fetcher
- logs
- report
- save file

Say:

- Metadata Signal Acquisition Array
- Metadata Void Cartography Engine
- Placeholder Suppression Firewall
- Source Intelligence Layer
- Series Continuity Engine
- Localization Policy Engine
- Context-Aware Metadata Reconstruction Core
- Non-Destructive Governance Layer
- Operational Metadata Ledger
- Metadata Command Center
- Contribution-Ready Distribution Rail

## Domain Rule

A component is not important because it exists. It is important because it
subdues a specific failure mode.

Examples:

- Signal Acquisition controls weak source context.
- Void Detection controls invisible metadata debt.
- Placeholder Suppression controls TBA/TBD debris.
- Continuity Assembly controls inconsistent episode naming.
- Reconstruction Core controls localized record quality.
- Governance controls accidental overwrites.
- Command Center controls operational blindness.
- Distribution Rail controls downstream delivery.

## Tone Rule

Keep the claims technically honest. Do not invent integrations, partnerships,
users, or unsupported language guarantees.

But never make a capability walk on stage unnamed.

Every feature should have:

- a role;
- a domain;
- a quality boundary;
- a place in the recovery system.

The final feeling should be controlled power, not utility drawer.
